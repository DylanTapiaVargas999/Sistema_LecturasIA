Visión General
El Sistema Educativo de Lectura Personalizada es una plataforma web innovadora que utiliza inteligencia artificial para personalizar la experiencia de aprendizaje de lectura. El sistema atiende a cuatro tipos de usuarios principales (estudiantes, docentes, padres y administradores) proporcionando funcionalidades específicas según sus roles y necesidades.

Restricciones
El sistema debe desarrollarse utilizando .net asp web api más react , según los lineamientos establecidos por el equipo de desarrollo.
La base de datos a utilizar será SQL Database, limitando la elección de otros gestores.
La aplicación debe ser web responsiva, sin contemplar en esta primera versión aplicaciones nativas móviles.
Toda la comunicación entre cliente y servidor debe realizarse sobre HTTPS.
El almacenamiento de contraseñas debe realizarse utilizando algoritmos de hashing y cifrado estándar (ej. SHA-256, bcrypt).
Se deben cumplir las regulaciones de protección de datos personales aplicables en el contexto educativo (ej. Ley de Protección de Datos Personales en Perú o GDPR en caso de usuarios internacionales).
El sistema deberá funcionar únicamente en las últimas versiones estables de Google Chrome y Mozilla Firefox.
No se garantiza soporte para navegadores obsoletos (ej. Internet Explorer) ni versiones antiguas de Edge o Safari.

Fuente: Elaboración propia.
Nota: El diagrama de casos de uso modela la funcionalidad completa del sistema desde la perspectiva de sus tres actores principales: Estudiante, Docente y Administrador.
El Estudiante es el actor central del sistema, ya que gestiona sus preferencias de contenido, accede a su biblioteca personal de lecturas, genera textos adaptados con IA, realiza evaluaciones inteligentes y recibe retroalimentación automatizada a través de la interfaz de lectura.
El Docente gestiona la creación y administración de aulas, el vínculo con los estudiantes, y visualiza los análisis educativos que resumen el desempeño y progreso del grupo. Además, puede acceder a la configuración de su cuenta y utilizar el sistema de ayuda interactiva.
El Administrador se encarga de las tareas de gestión general del sistema, como la administración de usuarios, la supervisión de las operaciones de la plataforma y el soporte mediante la funcionalidad de ayuda y tutoriales.

ATRIBUTOS DE CALIDAD DEL SOFTWARE
A continuación, se describen los atributos de calidad más críticos a través de escenarios específicos. Estos escenarios definen cómo debe comportarse el sistema en situaciones clave para garantizar su éxito.
Escenario de Seguridad
Este escenario se enfoca en proteger la información sensible de los estudiantes y prevenir el acceso no autorizado a los datos.
Fuente de Estímulo: Un usuario externo o interno malintencionado (Ej: un estudiante autenticado en el sistema).
Estímulo: El estudiante, que está correctamente autenticado, intenta acceder a los datos de otro compañero modificando el ID en la URL. Por ejemplo, cambia .../Progreso/Lecturas/12 (su propio ID) por .../Progreso/Lecturas/13 (el ID de otro estudiante).
Artefacto: El subsistema de Autorización (ASP.NET Core Identity), los Controladores (ProgresoController) y los Servicios de Negocio (ProgresoService).
Entorno: El sistema se encuentra en operación normal.
Respuesta: El sistema debe verificar que el ID del usuario autenticado coincide con el ID del recurso que está solicitando. Al detectar que no coinciden, debe denegar el acceso, registrar el intento de acceso no autorizado y redirigir al usuario a una página de error de "Acceso Denegado" (Error 403) o a su propio panel de control.
Medida de Re|spuesta: La denegación de acceso debe ocurrir en el 100% de los intentos. La información del otro estudiante nunca debe ser mostrada. El intento debe ser registrado en los logs del sistema en menos de 500 milisegundos.






Escenario de Usabilidad
Este escenario garantiza que las tareas comunes para los usuarios clave, como los docentes, sean eficientes e intuitivas.
Fuente de Estímulo: Un docente con conocimientos informáticos básicos.
Estímulo: El docente desea crear una nueva lectura, asignarle un título, copiar y pegar el contenido del texto, y adjuntar dos imágenes desde su computador.
Artefacto: La interfaz de usuario para la creación de lecturas (CreateReading.cshtml) y el ReadingsController.
Entorno: El docente está utilizando el sistema durante su horario de planificación.
Respuesta: El sistema presenta un formulario claro y bien estructurado. Los campos son intuitivos, y el proceso de subida de imágenes es simple (arrastrar y soltar o un botón de selección). Al finalizar, el sistema muestra un mensaje de confirmación claro y redirige al docente a la lista de lecturas.
Medida de Respuesta: Un docente nuevo, después de una única sesión de capacitación de 10 minutos, debe ser capaz de completar esta tarea en menos de 3 minutos.










Escenario de Adaptabilidad
Este escenario asegura que el sistema pueda ser modificado o extendido fácilmente en el futuro para incorporar nuevos requisitos.
Fuente de Estímulo: Un desarrollador del equipo de mantenimiento.
Estímulo: La dirección de la institución solicita un nuevo tipo de reporte para los administradores: "un reporte que muestre el tiempo promedio de lectura por nivel de grado".
Artefacto: El Subsistema de Lógica de Negocio (Servicios) y el Subsistema de Presentación (Controladores y Vistas).
Entorno: Durante una iteración de la fase de Construcción.
Respuesta: El desarrollador crea un nuevo método en un servicio existente (ReportService) para calcular la nueva métrica. Luego, añade una nueva acción en el AdminController y una nueva vista para mostrar el reporte. Los cambios no deben afectar la lógica de creación de lecturas, asignaciones o quizzes.
Medida de Respuesta: Un desarrollador debe ser capaz de implementar este nuevo reporte y sus pruebas correspondientes en menos de 12 horas de trabajo, sin necesidad de modificar el esquema de la base de datos.










Escenario de Disponibilidad
Este escenario se enfoca en la capacidad del sistema para permanecer operativo y recuperarse de fallos de software.
Fuente de Estímulo: Un fallo interno del sistema.
Estímulo: El proceso de la aplicación ASP.NET Core (Kestrel) se detiene inesperadamente debido a una excepción no controlada o un problema de memoria mientras 50 usuarios están utilizando el sistema.
Artefacto: El servidor de aplicación, el proceso Kestrel y el gestor de procesos del sistema operativo (ej. systemd en Ubuntu).
Entorno: El sistema está bajo carga normal durante el horario escolar.
Respuesta: El gestor de procesos del servidor (systemd) detecta que el proceso de la aplicación ha fallado y lo reinicia automáticamente. Durante el breve tiempo de reinicio, Nginx podría mostrar una página de error "502 Bad Gateway" a los nuevos usuarios, pero el sistema se recupera sin intervención manual.
Medida de Respuesta: El sistema debe estar completamente funcional y disponible de nuevo en menos de 90 segundos desde el momento del fallo. La disponibilidad total del sistema debe ser del 99.5% medida mensualmente.









Otro Escenario
Rendimiento
Este escenario crítico garantiza que el sistema sea rápido y responsivo bajo una carga de trabajo realista
Fuente de Estímulo: Un grupo de usuarios concurrentes.
Estímulo: 100 estudiantes finalizan una lectura y acceden al quiz asociado a ella de forma simultánea (por ejemplo, al final de una clase).
Artefacto: El servidor de aplicación (CPU, RAM), el QuizController, el QuizGenerationService y la base de datos (consultas para generar y mostrar el quiz).
Entorno: Pico de carga de trabajo al final de una actividad de lectura programada.
Respuesta: El sistema procesa todas las solicitudes, genera (si es necesario) los quizzes personalizados y los muestra a cada uno de los 100 estudiantes.
Medida de Respuesta: El tiempo de carga de la primera pregunta del quiz para el 95% de los estudiantes (percentil 95) debe ser inferior a 4 segundos. El uso del CPU en el servidor de aplicación no debe superar el 80% durante este pico.
